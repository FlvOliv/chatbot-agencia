Deploy da Malu em produção

Sequência completa pra tirar a Malu do laptop e colocar pra atender clientes reais. Cada caixa marca progresso e salva no navegador — pode fechar e voltar.

Progresso— 0 de 0 passos

Antes de começar

  • Tudo localmente verde: pytest tests/ -v passa, npm run build no painel passa
  • main do GitHub atualizada com o painel mergeado
  • Contas criadas (ou prontas pra criar): Supabase, Upstash, Railway, Vercel — todas grátis pra começar
  • Privacy Policy publicada em algum URL (use docs/PRIVACIDADE.html via GitHub Pages se ainda não tem)
  • Tokens da Meta em mãos (WA_TOKEN permanente, WA_APP_SECRET, Phone Number ID, Verify Token)
1
Supabase · Postgres na nuvem
~10 min
▾
Criar projeto Supabase

Abre supabase.com · Start your project · login com GitHub · New project.

Preenche:

  • Name: malu-bot-producao
  • Database Password: clica em "Generate" e salva em local seguro (1Password, gerenciador, etc.)
  • Region: South America (São Paulo)
  • Pricing: Free

Aguarda ~2 min provisionar.

Pegar a connection string

Projeto criado → Settings (engrenagem) → Database → Connection string → URI → modo Session pooler (porta 5432).

Copia. Vai parecer:

postgresql://postgres.xxxxx:[YOUR-PASSWORD]@aws-0-sa-east-1.pooler.supabase.com:5432/postgres

Substitui [YOUR-PASSWORD] pela senha salva no passo 1.1, e troca postgresql:// por postgresql+asyncpg://:

postgresql+asyncpg://postgres.xxxxx:SUASENHA@aws-0-sa-east-1.pooler.supabase.com:5432/postgres

Esse é o valor que vai pra DATABASE_URL no Railway (etapa 3).

Habilitar pgcrypto

A migration 0001_initial usa gen_random_uuid() que vem de pgcrypto.

Supabase → Database → Extensions → procura pgcrypto → clica em Enable.

2
Upstash · Redis serverless
~5 min
▾
Criar database Redis

Abre upstash.com · sign up com GitHub · Create Database.

Preenche:

  • Name: malu-bot-prod
  • Type: Regional (mais barato)
  • Region: sa-east-1 (São Paulo)
  • TLS: ✅ Enabled (default — não desabilita)
Copiar a TLS URL

Database criado → aba Details → copia o UPSTASH_REDIS_TLS_URL.

Vai parecer:

rediss://default:xxxxxxx@xxxxx.upstash.io:6379

Esse vai pra REDIS_URL no Railway. Repara no rediss:// (com SS — TLS).

3
Railway · Service web (FastAPI)
~15 min
▾
Criar projeto Railway

Abre railway.app · login com GitHub · New Project → Deploy from GitHub repo.

Autoriza acesso ao repo gustavomarcelloprf/chatbot-agencia e seleciona. Railway detecta o Dockerfile e começa o build (vai falhar por enquanto, só sem as vars).

Adicionar variáveis de ambiente

No painel do projeto → Variables → cola tudo de uma vez (botão "Raw editor"):

WA_TOKEN=<token permanente da Meta> WA_PHONE_ID=<phone number id> WA_VERIFY_TOKEN=<sua string aleatória> WA_APP_SECRET=<app secret da Meta> GEMINI_API_KEY=AIza... GEMINI_MODEL=gemini-2.5-flash GROQ_API_KEY=gsk_... GROQ_MODEL=llama-3.3-70b-versatile AI_PRIMARY=gemini AI_FALLBACK=auto DATABASE_URL=postgresql+asyncpg://postgres.xxxxx:SENHA@aws-0-sa-east-1.pooler.supabase.com:5432/postgres REDIS_URL=rediss://default:xxxxx@xxxxx.upstash.io:6379 LUCIANA_PHONE=5511... BUSINESS_HOURS_START=9 BUSINESS_HOURS_END=18 APP_ENV=production LOG_LEVEL=INFO SESSION_TTL_SECONDS=86400 CRM_API_KEY=<gerar nova chave> CRM_CORS_ORIGINS=https://placeholder.vercel.app WEB_CONCURRENCY=2 WORKER_CONCURRENCY=2

Gera a CRM_API_KEY com:

python -c "import secrets; print(secrets.token_urlsafe(32))"

CRM_CORS_ORIGINS fica com placeholder agora — atualiza na etapa 6 com a URL real do Vercel.

Confirmar build verde + gerar domínio público

Aba Deployments → aguarda 3-5 min até aparecer Active.

Aba Settings → Networking → Generate Domain → você ganha algo tipo malu-bot.up.railway.app.

Testar /health

Do terminal:

curl https://malu-bot.up.railway.app/health

Esperado: {"status":"ok",...}

⚠️ Se voltar 500 ou timeout, abre Deployments → Logs. Erros comuns: DATABASE_URL sem +asyncpg, ou Supabase ainda inicializando.
Testar endpoint de insights
curl -H "X-API-Key: $CRM_API_KEY" "https://malu-bot.up.railway.app/api/dashboard/insights?days=7"

Esperado: JSON com conversations_per_day, top_destinations, etc. (vazio com banco novo, mas estrutura correta).

4
Railway · Service worker (Celery)
~5 min
▾
Criar segundo service no mesmo projeto

No projeto Railway → botão + New → GitHub Repo → mesmo repo.

Renomeia pra malu-worker em Settings → Service Name.

Custom Start Command

Settings → Deploy → Custom Start Command:

./scripts/start_worker.sh
Reference variables do service web

Em Variables do worker, clica Add Reference e seleciona todas as vars do service web. Importantes: DATABASE_URL, REDIS_URL, WA_TOKEN, LUCIANA_PHONE, GEMINI_API_KEY, GROQ_API_KEY.

Confirmar worker rodando

Aba Deployments → Logs → deve aparecer:

celery@malu-worker ready.

Worker não precisa de domínio público — só consome a fila Redis.

5
Meta · Atualizar webhook
~5 min
▾
Trocar URL do webhook

Abre developers.facebook.com · seu app · WhatsApp → Configuration · em Webhook clica Edit.

  • Callback URL: https://malu-bot.up.railway.app/webhook
  • Verify Token: mesmo de WA_VERIFY_TOKEN

Clica Verify and save — deve ficar verde.

Garantir que `messages` está assinado

Em Webhook fields, marca messages (se já está marcado, deixa).

Confirmar Publish (Live mode) se ainda não fez

Se o app ainda está em Development, sai pra Live agora — caso contrário só números na allowed list recebem resposta.

Topo da página do app → toggle App Mode → Live. Precisa ter Privacy Policy URL configurada (use a página estática que geramos).

6
Vercel · Painel de insights
~15 min
▾
Import project no Vercel

Abre vercel.com · Add New → Project · importa o repo gustavomarcelloprf/chatbot-agencia.

Config:

  • Framework Preset: Next.js (auto-detectado)
  • Root Directory: crm/ ← importante!
  • Node.js Version: 20.x ou 22.x
Adicionar Environment Variables

Em Project Settings → Environment Variables:

NEXT_PUBLIC_API_BASE_URL=https://malu-bot.up.railway.app CRM_API_KEY=<mesma do Railway> NEXT_PUBLIC_LUCIANA_PHONE=5511... NEXT_PUBLIC_AI_PRIMARY=gemini NEXT_PUBLIC_AI_FALLBACK=groq NEXT_PUBLIC_BUSINESS_HOURS_START=9 NEXT_PUBLIC_BUSINESS_HOURS_END=18
Cuidado: CRM_API_KEY NÃO leva prefixo NEXT_PUBLIC_ — ela é server-side only. Se vazar pro client, qualquer um lê leads.

Marca todas como Production + Preview (não Development).

Deploy

Clica Deploy. Aguarda ~2-3 min. Copia o domínio gerado (algo tipo https://chatbot-agencia.vercel.app).

Atualizar CORS no Railway

Volta no Railway → service web → Variables → edita CRM_CORS_ORIGINS:

CRM_CORS_ORIGINS=https://chatbot-agencia.vercel.app

Save → redeploy automático.

Validar painel

Abre https://chatbot-agencia.vercel.app no celular (de preferência) e no desktop.

Deve ver: dashboard com 4 KPIs, gráficos, leads (vazio se banco novo), configurações.

⚠️ Se algum bloco mostrar "—": abre DevTools → Network → procura erro de CORS ou 401. Geralmente é CRM_API_KEY divergente ou CRM_CORS_ORIGINS esquecido.
7
Validação end-to-end
~10 min
▾
Mandar mensagem real pelo WhatsApp

Do seu celular, manda uma mensagem qualquer pro número da Malu. Em <10s deve chegar resposta.

Confere logs no Railway (service web → Deployments → Logs):

INFO malu — incoming from 5511...: oi INFO httpx — POST .../messages "HTTP/1.1 200 OK" INFO malu — whatsapp send_message OK
⚠️ Se não responder e estiver em modo Development: o seu número precisa estar na "Allowed phone number list" da Meta. Se já está Live, deveria funcionar pra qualquer número.
Completar uma cotação até o briefing

Continua a conversa fingindo cliente: passa destino, datas, número de pessoas. A Malu deve gerar o bloco ## Resumo da Solicitação de Cotação no fim.

Confirmar notificação chegou na Lu (você por enquanto)

Você (no número LUCIANA_PHONE) deve receber uma mensagem da Malu com o briefing.

Conferir que apareceu no painel

Abre o painel Vercel no celular → dashboard → o lead deve aparecer em "Últimos leads", com temperatura. Click → abre o briefing.

Backup do .env de produção

Copia todas as variáveis do Railway num arquivo seguro (1Password, gerenciador de senhas, ou arquivo cifrado local). Sem isso, recriar o deploy depois de perder a conta vira um inferno.

Bom deploy 🚢

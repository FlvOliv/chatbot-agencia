  • Histórico de conversa no Redis (TTL 24h)
  • Tabela clientes + captura do profile.name do WhatsApp
  • Tabela reservas + fluxo "cliente novo vs existente" (1/2 pelo cliente)
  • Estado de sessão (awaiting_intent · transferred) em Redis
  • Comando /sair encerra sessão limpa
  • Lembretes de inatividade (15min · 5h · 23h) via Celery, idempotentes e canceláveis
  • Briefing parsing: temperatura, WhatsApp do cliente, nome preferido
  • Notificação para Lu via WhatsApp (briefing novo & cliente retornante)
  • Worker Celery: follow-up + relatório diário 8h
  • APIs REST /api/* com autenticação X-API-Key e CORS configurado

CRM web (Next.js · TypeScript · shadcn/ui)

Arquitetura

Sistema de 4 camadas. Tudo async. Estado da conversa vive no Redis; persistência durável no Postgres.

┌─────────────────────────────────────────────────────────────────┐ │ Cliente (WhatsApp) │ └──────────────────────────────┬──────────────────────────────────┘ │ texto + áudio (futuro) ▼ ┌─────────────────────────────────────────────────────────────────┐ │ Meta WhatsApp Cloud API (oficial) │ │ webhook POST /webhook ◄► outbound /messages │ └──────────────────────────────┬──────────────────────────────────┘ │ HTTPS + HMAC-SHA256 ▼ ┌─────────────────────────────────────────────────────────────────┐ │ Backend Malu (FastAPI async) │ │ ┌────────────┐ ┌────────────┐ ┌─────────────┐ ┌──────────┐ │ │ │ Webhook │ │ handle_msg │ │ Router IA │ │ Briefing │ │ │ │ + HMAC │ │ pipeline │ │ Gemini/Groq │ │ + Lu │ │ │ └────────────┘ └────────────┘ └─────────────┘ └──────────┘ │ │ ┌────────────────────────────────────────────────────────────┐ │ │ │ REST /api/* (X-API-Key) → leads · conversations · ... │ │ │ └────────────────────────────────────────────────────────────┘ │ └──────┬──────────────────────┬──────────────────────┬────────────┘ │ │ │ ▼ ▼ ▼ ┌─────────────┐ ┌───────────────┐ ┌─────────────────┐ │ Postgres │ │ Redis │ │ Celery worker │ │ (Supabase) │ │ (Upstash) │ │ + Beat schedule │ │ leads │ │ session 24h │ │ lembretes 15m/ │ │ clientes │ │ state │ │ 5h/23h │ │ reservas │ │ reminders ids │ │ daily report 8h │ │ convs │ │ │ │ │ └─────────────┘ └───────────────┘ └─────────────────┘ ▲ │ /api/* via X-API-Key │ ┌─────────────────────────────────────────────────────────────────┐ │ CRM web (Next.js · server components) │ │ dashboard · leads · conversas · reservas · configurações │ └─────────────────────────────────────────────────────────────────┘

Pipeline de uma mensagem

1. Meta envia POST /webhook
2. App valida HMAC (rejeita 401 se assinatura inválida)
3. Resposta 200 imediata; processamento em background
4. Comando /sair?  → encerra sessão, cancela lembretes, manda despedida
5. Estado "transferred"? → silêncio total (Lu assumiu)
6. get_or_create_cliente(phone, profile_name) no Postgres
7. Estado "awaiting_intent"?
     ├── "1" / "reserva" → notify Lu, marca transferred, sai
     ├── "2" / "nova"    → limpa estado, segue pra IA
     └── ambíguo         → repete pergunta
8. Primeira mensagem + reserva ativa? → pergunta 1/2, marca state
9. IA: route_and_ask(history, customer_context)
     - Gemini primário (5 RPM free) → se falhar, Groq automático
10. Salva histórico (Redis), envia resposta, audit log
11. Briefing detectado? → save_lead + update_preferred_name + notify Lu
12. Agenda lembretes de inatividade (15m / 5h / 23h)

O que falta refinar/adicionar (detalhado)

Refinos no backend Malu

  • Detector de tipo de mensagem (áudio/imagem/vídeo/sticker) com resposta educada
  • Transcrição de áudio via Gemini Audio (versão completa, V2)
  • Rate limit por phone (anti-spam — cliente disparando 50 msgs por minuto)
  • Endpoint POST /api/conversations/{phone}/takeover pra Lu marcar conversa como transferida pelo CRM
  • Endpoint POST /api/conversations/{phone}/send pra Lu mandar mensagem pelo CRM e sair pelo número da Malu
  • Telemetria leve: tempo médio de resposta, taxa de fallback Gemini→Groq, etc.

Refinos no CRM

  • Conectar com API real (em andamento)
  • Autenticação real — hoje só tem X-API-Key compartilhada. Auth.js (Google login) quando virar multi-tenant
  • Realtime nas conversas (Server-Sent Events ou polling 5s)
  • Dashboard com gráficos (conversas/dia, taxa conversão, destinos top, etc.) — base pro plano "Insights"
  • Mobile-friendly (Lu vai usar pelo celular boa parte do tempo)

Decisões tomadas (registro)

  • IA: Gemini 2.5 Flash primário · Groq Llama 3.3 70B fallback automático (auto). Permite trocar via .env sem código.
  • Persona: Lu (não Luciana) · regra absoluta anti-preço · seção 22 do prompt blinda jailbreak.
  • Modelo de negócio: começar single-tenant focado na Lu. Multi-tenant fica pra v2 com 2+ clientes validados.
  • CRM: monorepo via git worktree (subdir crm/). Backend e frontend lado a lado sem conflito.
  • Stack DB: Postgres no Supabase + Redis no Upstash (descartamos Docker local em produção).
  • Chatwoot: descartado por agora. CRM próprio cobre o caso.
  • Agente de IA buscando preços: fora do MVP. Volta como produto pago plano 3 no futuro.
